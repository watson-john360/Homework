Gitflow and Phabricator

Phabricator utilizes a different workflow from Gitflow, as Phabricator is designed specifically for Trunk Based Development.

Gitflow

The Gitflow process puts all code changes in separate branches, which are then reviewed upon request and merged once the feature is fully completed. While it appears to be similar to trunk based development, branch-based development must flow through a single reviewer (or group of reviewers) before the code is committed to the trunk. Branches are usually larger in scope and longer in duration, as compared to trunk-based development.

The advantage of Gitflow is that code quality is strictly controlled by smaller group(s) in charge of reviewing and approving the commit requests. While this can be especially helpful for open-source projects and projects where developer trust and code quality are concerns, the downside is that the reviewers can easily become a choke-point in the development process. Further slowdowns can occur when multiple larger commits are merged at the same time, which can easily cause merge conflicts.

Phabricator

The Phabricator workflow puts code in a review state before it is merged; code cannot be merged until it has been reviewed. The need for reversions is minimized as code is reviewed/revised as it is being developed, helping keep the trunk clean and allowing for greater control over the merges to the trunk. Most importantly, issues are kept non-blocking when they are caught and fixed in the review stage.

As Uber ATG is not open source, trusts their developers, and requires a quick, continual integration and testing of code, the Phabricator workflow to allow for faster code deployment while mitigating the risks of Trunk Based Development.
